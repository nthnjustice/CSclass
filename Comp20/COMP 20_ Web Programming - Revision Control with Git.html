Distributed Revision Control Software

  • Why?
  • Source code management
  • Distributed - many developers can work on a given project from different locations and networks

Git

  • A distributed revision control system
  • Extremely fast and efficient
  • Lightweight
  • Largely local-based, not dependent on a remote network (unlike Subversion)

Basic Git Commands

  1. git clone GIT_URL - Download a copy of a repository
  2. git pull - Update your repository with latest changes from remote
  3. git add - Add file(s) to your staging area
  4. git commit - Record your snapshot into your history; "bless" your work
    • Important: each commit is given a unique SHA-1 hash identifier
  5. git push - Upload your changes to the remote repository (e.g., GitHub)
  6. git log - Show commit logs

Reverting to a Previous Commit

  • git reset --hard HASH_OF_THE_COMMIT
  • Source: http://stackoverflow.com/questions/4114095/revert-to-previous-git-commit

Branching

  • Why: "diverge from the main line of development and continue to do work without messing with that main line" (source: http://git-scm.com/book/en/Git-Branching)
  • git branch ...
  • git checkout ...
  • You will need to use branching in order to publish to GitHub pages (i.e., gh-pages). https://help.github.com/articles/creating-project-pages-manually

GitHub

  • A website: https://github.com/
  • Not to be confused with Git itself
  • Web-based hosting of Git projects
  • Fosters even greater collaboration!
  • Fork popular projects
  • Social

Advice

  • Almost never a good idea to work directly off of the master branch
  • Worse comes to worst if you are having problems with a Git repo, delete the entire folder, re-clone it from GitHub and work off it that
